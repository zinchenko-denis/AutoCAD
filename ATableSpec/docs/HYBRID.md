# Гибрид: C#-плагин AutoCAD ↔ Python-движок (dxf_spec)

Архитектура (C): мозги — в Python (`dxf_spec`, сопровождается легко), тонкий
C#-плагин в AutoCAD собирает блоки из живого чертежа, отдаёт их движку и рисует
результат таблицей `AcDbTable` обратно в DWG. Логика спецификаций НЕ дублируется
в C#.

```
  AutoCAD (живой DWG)
      │  команда ATSPEC, выбор блоков (фильтр INSERT)
      ▼
  C#-плагин (тонкий)                 ← версионно-зависимая часть
      │  JSON: массив записей блоков (stdin)
      ▼
  dxf_spec.exe  (Python, заморожен PyInstaller)   ← версионно-НЕзависимая часть
      │  JSON: спецификации (stdout)
      ▼
  C#-плагин рисует AcDbTable в modelspace
```

## Контракт обмена (стабилен, не зависит от версии AutoCAD)

**Вход движка** — то, что C# собирает из чертежа и шлёт в stdin
(`dxf_spec.exe --json -c mapping.yaml`):
```json
[
  { "name": "*U96", "layer": "RF-ригеля",
    "x": 606217.65, "y": 471422.40, "rotation": 270.0,
    "xscale": 1.0, "yscale": 1.0,
    "attributes": { "ИМЯ": "Р01.2", "ПРОФ": "17_06_01", "ДЛИНА": "540.50" } }
]
```
(полный пример — `records.sample.json`; то же отдаёт `--emit-records` из DXF.)

**Выход движка** — stdout, C# его разбирает и рисует:
```json
{ "ok": true, "blocks_in": 924, "elements": 900, "skipped": 24,
  "reports": [
    { "name": "заполнения", "title": "Спецификация заполнений",
      "columns": ["type","mark","width","height","кол_во"],
      "rows": [ {"type":"заполнение","mark":"ст2","width":670.0,"height":1170.0,"кол_во":54} ],
      "total_qty": 162, "total_length_mm": null } ] }
```
C# рисует таблицу из `columns` + `rows` выбранного отчёта. Всё, что меняется
под проект (какие слои, теги, отчёты), живёт в `mapping.yaml`, а не в C#.

Тот же контракт проверяется без AutoCAD:
```bash
python dxf_spec.py CHERTEZH.dxf -c mapping.yaml --emit-records > records.json
cat records.json | python dxf_spec.py -c mapping.yaml --json > specs.json
```

### Действие `report` — «свой отчёт» (аналог Шаблона отчёта СПДС)

Помимо пресетов (`run`) и одиночного запроса (`query`), движок принимает
`action:"report"` с определением отчёта, где **каждый столбец задаётся
выражением** над полями блока (как в построителе выражений СПДС). Команда
плагина — `ATSPECREPORT`. Логика — модуль `atspec_report.py` (чистая, без ezdxf;
покрыта `test_atspec_report.py`).

Вход:
```json
{ "blocks": [ ... ], "action": "report",
  "report": {
    "title": "СПЕЦИФИКАЦИЯ ЭЛЕМЕНТОВ",
    "header": ["№ п/п","НАИМЕНОВАНИЕ","Артикул","Длина, мм","Колич.","Ед. изм."],
    "templates": [
      { "filter":   [ {"field":"Слой","op":"=","value":"RF-стойки"} ],
        "columns":  ["=row-1","=Object.«ИМЯ»","=Object.Name","=Object.«Длина»","=Count","=«шт.»"],
        "group_by": 1,
        "sort_by":  [1, "asc"] }
    ] } }
```
Выход — `{ "ok": true, "report": { "title", "header", "rows": [[...], ...] } }`;
C# рисует `AcDbTable` позиционно по `header` + `rows`.

Поддержка выражений: `Object.«поле»` / `Object.Name` / `Object.«Слой»`,
арифметика `±n`/`*`/`/`, литералы `=«…»`, конкатенация `+`, `Count`, `Sum(...)`,
`Iff(cond,a,b)`, сравнения, автонумерация `row`. Несколько `templates` в одном
отчёте дают производные строки (крышка+прижим из стойки) — их результат
конкатенируется. Это путь по СЫРЫМ записям блоков (любой атрибут/слой), он НЕ
зависит от проекции `mapping.yaml`.

## Состав поставки (.bundle)

Один автозагружаемый бандл Autodesk — пользователь копирует одну папку:
```
ATableSpec.bundle/
  PackageContents.xml          ← маршрутизация по версии AutoCAD
  Contents/
    net48/ATableSpec.dll       ← плагин для AutoCAD 2013–2024 (.NET Framework)
    net8/ATableSpec.dll        ← плагин для AutoCAD 2025+ (.NET 8)
    engine/dxf_spec.exe        ← Python-движок (заморожен; один на все версии)
    engine/mapping.yaml        ← конфиг проекта
```

## Поддержка «всех версий AutoCAD» — честно

Единым *бинарником* все версии не покрыть из-за жёсткого разлома среды
выполнения:
- **AutoCAD 2013–2024** — управляемый рантайм **.NET Framework 4.x**.
- **AutoCAD 2025+** — Autodesk перевёл API на **.NET 8 (.NET Core)**.
Одна DLL не может целить и в .NET Framework, и в .NET 8 одновременно → нужно
**минимум две сборки** плагина. `PackageContents.xml` через
`SeriesMin/SeriesMax` отдаёт каждой версии AutoCAD свою DLL.

Что покрывает все версии без пересборки:
- **движок `dxf_spec.exe`** — отдельный процесс, среды не касается: один на всё;
- **контракт JSON** — стабилен;
- **механизм автозагрузки** бандла.

Внутри семейства .NET Framework одна DLL (`net48`, ссылки на сборки AutoCAD с
`Specific Version=False`, `Copy Local=False`) обычно грузится во всём диапазоне
2013–2024; при изменении сигнатур API в каком-то релизе туда добавляется
отдельная DLL — бандл это позволяет адресно.

Итог: «все версии» поддерживает **дистрибутив** (бандл), а скомпилированные DLL
добавляются по диапазонам по мере надобности. Для пилота собираем под версию
конструктора, архитектуру закладываем расширяемой.

## Установка для не-специалиста (черновик; финал — после фиксации версии)

1. Скопировать папку `ATableSpec.bundle` в:
   `%APPDATA%\Autodesk\ApplicationPlugins\`
   (полный путь обычно `C:\Users\<Имя>\AppData\Roaming\Autodesk\ApplicationPlugins\`).
2. Запустить AutoCAD. Бандл подхватится автоматически (NETLOAD вручную не нужен).
3. В чертеже набрать команду `ATSPEC`, выбрать блоки (или рамкой весь фасад),
   нажать Enter, указать точку вставки таблицы.

## Расширение под новые задачи (пилот → бой)

- Новая ведомость / другая группировка → новый блок в `reports` в `mapping.yaml`.
- Новый тип элемента / другой слой / другие теги → правки в `element_layers` /
  `field_map` там же.
- Принципиально новая задача (напр. раскрой с оптимизацией) → отдельный модуль
  Python поверх того же контракта; C# и установка не меняются.
